# F27 — Three-strategy recommendation (optimal / fastest-payback / long-term)

> **North-Star link:** The three strategy cards each carry a `monthly_saving` and a `break_even_month`,
> giving the user three honest frames on the same number — not a different engine, just three named
> pointers into the ladder already computed by F10/F11. The default CTA is **Optimal**, which is where
> Cloover's headline saving lives.

## 1. Intent (what & why)

Implement R-D (§0, §6.6): beyond the single optimiser pick, `recommend()` now selects **three named
strategies** over the F10 ladder and returns them in a `strategies` field on the `Recommendation`
response (§14.1). Each strategy is a `{ id, label, scenario: ScenarioResult, rationale, break_even_month }`
pointing at an existing `ScenarioResult` from `alternatives[]` — no new engine computation. The three
strategies are **Optimal** (full bundle / max monthly saving), **Fastest payback** (argmin
`break_even_month` among net-positive rungs), and **Long-term** (argmax `saving_after_payoff_eur`).
The F02 API contract gains a `strategies[]` field; F22 renders them as three Check24-style cards with
Optimal as the default CTA (§6.6, §9). The engine remains pure (zero I/O, zero DB).

## 2. Scope

**In scope**
- `select_strategies(ladder: list[ScenarioResult]) → dict[str, Strategy]` — pure function in the domain core (`apps/api/src/app/domain/`).
- Three `Strategy` objects: `optimal`, `fastest_payback`, `long_term` — each a named pointer + label + one-line `rationale` string + `break_even_month`.
- Contract extension: `Recommendation.strategies: Strategy[]` (three items) added to `specs/api/openapi.yaml` in the same PR (§14.1).
- Tie-handling: documented rules for when two strategies coincide (same `ScenarioResult`), including the case where all three collapse to the same rung.
- The `rationale` string is generated by the pure domain function (a deterministic template string); the LLM advisor (F16) may enrich it with prose but must not alter the numeric fields.

**Out of scope** (explicitly, to prevent creep)
- UI card rendering → **F22** (consumes `strategies[]` from the contract).
- The LLM prose paragraph within `rationale_md` → **F16** (receives the strategy set and elaborates prose-only).
- À-la-carte subset evaluation beyond the four nested ladder rungs → **stretch** (D3).
- A fourth strategy (e.g. "income-optimised") → not in §6.6; cut if proposed.
- Any engine I/O, DB call, or network call — this feature is pure domain (§1 domain invariants).

## 3. Functional requirements

| # | Requirement | Source (§) |
|---|-------------|-----------|
| R1 | `recommend()` returns a `strategies` field containing exactly three named strategies: `optimal`, `fastest_payback`, `long_term`. | §6.6, §14.1 |
| R2 | **Optimal** = the ladder rung (or full bundle) with the highest `monthly_saving_eur` among net-positive rungs; falls back to the rung with the least-negative saving if none is net-positive. Label example: *"Full bundle — best monthly outcome"* (§6.6). | §6.6 |
| R3 | **Fastest payback** = argmin `break_even_month` among rungs where `delta_net > 0` (i.e. the rung that reaches cumulative net ≥ 0 soonest). Least capital tied up; earliest profit (§6.6). | §6.6 |
| R4 | **Long-term** = argmax `saving_after_payoff_eur` (maximises lifetime value / decades impact, §6.6). | §6.6 |
| R5 | Each `Strategy` carries: `id` (`"optimal"|"fastest_payback"|"long_term"`), `label` (human string), `scenario: ScenarioResult` (reused from `alternatives[]`; no new compute), `rationale` (one-line deterministic template), `break_even_month` (copied from the selected scenario). | §6.6, §14.1 |
| R6 | If two or more strategies resolve to the same rung, they share the same `ScenarioResult` but retain distinct `id`, `label`, and `rationale`; the response still contains three objects. Ties are noted in `rationale` (e.g. *"Also the fastest-payback configuration"*). | §6.6 |
| R7 | The `select_strategies` function is **pure**: it consumes only the `ladder: list[ScenarioResult]` already produced by F10/F11; it imports no prices, no subsidies, no DB — per the domain invariants (specs/domain/savings-engine.spec.md §1). | §1 (domain invariants) |
| R8 | `openapi.yaml` is updated in the same PR with the `Strategy` schema object and `Recommendation.strategies: Strategy[]` (three items); the change is backwards-compatible (additive field). | §14.1, §6.6 |
| R9 | The optimiser's existing single-best pick (`Recommendation.best`) is unchanged — `strategies` is an additive field that does not replace it. | §6.4, §14.1 |

## 4. Data, formulas & sources

> N/A for fetched data — this is a pure selector over the already-computed ladder. All monetary
> quantities come from F10/F11; no new prices or constants are introduced here.

Selection logic (verbatim from §6.6, mapped to code):

```python
# ladder = list[ScenarioResult], ordered L1→L1+L2→L1+L2+L3→L1+L2+L3+L4
# (as produced by F10; some rungs may be absent if a layer has delta=0)

net_positive = [s for s in ladder if s.monthly_saving_eur > 0]

# Optimal: max monthly_saving; fallback to best available if none positive
optimal = max(ladder, key=lambda s: s.monthly_saving_eur)

# Fastest payback: argmin break_even_month among net-positive rungs only
#   (a negative-net rung never breaks even → excluded to avoid misleading frame)
fastest = min(net_positive, key=lambda s: s.break_even_month) if net_positive else optimal

# Long-term: argmax saving_after_payoff_eur (lifetime value)
long_term = max(ladder, key=lambda s: s.saving_after_payoff_eur)
```

Key fields consumed from `ScenarioResult` (all produced by F10/F11, no new compute):

| Field | Produced by | Used in strategy |
|---|---|---|
| `monthly_saving_eur` (= Σ delta_net) | F10/F11 | Optimal selection |
| `break_even_month` | F11 | Fastest payback selection |
| `saving_after_payoff_eur` (= Σ delta_gross) | F10/F11 | Long-term selection |

## 5. Contract surface

`contract_impact: extends` — `specs/api/openapi.yaml` is updated in the same PR.

New schema object:
```yaml
Strategy:
  type: object
  required: [id, label, scenario, rationale, break_even_month]
  properties:
    id:
      type: string
      enum: [optimal, fastest_payback, long_term]
    label:
      type: string
    scenario:
      $ref: '#/components/schemas/ScenarioResult'
    rationale:
      type: string
      description: One-line deterministic rationale; LLM may extend prose via F16
    break_even_month:
      type: integer
```

Changed field on `Recommendation`:
```yaml
strategies:
  type: array
  items:
    $ref: '#/components/schemas/Strategy'
  minItems: 3
  maxItems: 3
```

Backwards-compatible: additive field; existing consumers that ignore `strategies` are unaffected.

## 6. Acceptance criteria (testable — these become the tests)

- [ ] **AC1** — Given the `V_WORKED_LADDER` fixture (F03), when `select_strategies(ladder)` runs, then `optimal.scenario` is the full-bundle rung (all four layers, `monthly_saving_eur ≈ +€120`), and `optimal.break_even_month` matches the full-bundle engine output.
- [ ] **AC2** — Given the same fixture, when `select_strategies` runs, then `fastest_payback.scenario` is the rung with the lowest `break_even_month` among rungs where `monthly_saving_eur > 0` (expected: the heat-pump rung at +€20/mo or the EV rung, depending on engine output).
- [ ] **AC3** — Given the same fixture, when `select_strategies` runs, then `long_term.scenario` is the full-bundle rung (highest `saving_after_payoff_eur ≈ €364/mo`).
- [ ] **AC4 (tie)** — Given a ladder where only one rung is net-positive, when `select_strategies` runs, then all three strategies reference the same `ScenarioResult` but carry distinct `id` values and each `rationale` notes the tie.
- [ ] **AC5 (no-positive edge)** — Given a ladder where all rungs are net-negative (e.g. very high capex vs saving), when `select_strategies` runs, then `fastest_payback` falls back to `optimal` (no crash), and each strategy's `rationale` honestly reflects the negative net saving.
- [ ] **AC6 (purity)** — Given the `select_strategies` function source, when inspected, then it imports no prices, DB modules, or network calls; its only input is `list[ScenarioResult]`.
- [ ] **AC7 (contract)** — Given a `/recommend` API response, when parsed against the updated `openapi.yaml`, then `strategies` is present with exactly three items, each with a non-null `id`, `label`, `rationale`, and a `ScenarioResult` matching a rung in `alternatives[]`.

## 9. Risks & mitigations

| Risk | Mitigation |
|------|-----------|
| Three-strategy cards confuse user | Clearly labelled cards (§6.6, §15); Optimal is the default CTA; others are compare/upsell context. |
| `fastest_payback` is net-negative (no rung breaks even) | AC5 guards this; fallback to `optimal` with an honest rationale noting no rung is currently profitable during financing. |
| `long_term` coincides with `optimal` for most inputs | Tie rule (R6 + AC4): distinct objects with shared scenario; `rationale` notes the coincidence. Informative, not confusing. |
| LLM alters numeric fields in `rationale_md` | F16 spec enforces "prose only, no compute" + number-assertion guard (§15); `Strategy.rationale` and the numeric fields in `ScenarioResult` are immutable after engine output. |
| Contract bump blocks parallel FE work | `strategies[]` is an additive field (backwards-compatible); F22 can mock it immediately from the `V_WORKED_LADDER` fixture (§8). |
| Engine serial-owner bottleneck (Lukas owns F05–F11 chain) | F27 logic is a thin selector (~30 lines of pure Python) layered on top of the already-computed ladder — minimal additional load (§6 Backlog risk). |
